# API Contract

Single source of truth for backendŌåöfrontend integration. `backend/IMPLEMENTATION.md` and `frontend/IMPLEMENTATION.md` both reference this instead of restating request/response shapes ŌĆö update it first if the contract changes, then update whichever stack plan is affected.

Base URL (dev): `http://localhost:8000/api`

Built on a DRF viewset behind a `DefaultRouter`, exposing only `list`/`create`/`retrieve` ŌĆö the frontend never edits or deletes a task, so those verbs aren't exposed. `Task` uses a `HyperlinkedModelSerializer` (every task carries a `url` to itself); nested `steps` are plain, unlinked objects ŌĆö they have no endpoint of their own. `GET /api/` is the DRF browsable API root.

## `POST /api/tasks/`

Submit a task for the agent to process. The prompt is routed through `AgentController`, not saved as-is ŌĆö `result` and `steps` are populated by tool execution and are **read-only** on the client side (rejected/ignored if sent).

Request:

```json
{ "prompt": "What is 15 + 27?" }
```

Response `201 Created`:

```json
{
  "url": "http://localhost:8000/api/tasks/1/",
  "id": 1,
  "prompt": "What is 15 + 27?",
  "result": "42",
  "created_at": "2026-07-09T12:00:00Z",
  "steps": [
    { "step_number": 1, "description": "Received input \"What is 15 + 27?\"", "tool_name": null, "timestamp": "2026-07-09T12:00:00Z" },
    { "step_number": 2, "description": "Selected tool: CalculatorTool", "tool_name": "CalculatorTool", "timestamp": "2026-07-09T12:00:00Z" },
    { "step_number": 3, "description": "Tool result: 42", "tool_name": "CalculatorTool", "timestamp": "2026-07-09T12:00:00Z" },
    { "step_number": 4, "description": "Returning result to user", "tool_name": null, "timestamp": "2026-07-09T12:00:00Z" }
  ]
}
```

Response `400 Bad Request` (empty, missing, or whitespace-only prompt) ŌĆö standard DRF field-error shape:

```json
{ "prompt": ["This field may not be blank."] }
```

## `GET /api/tasks/`

List past tasks for the history view, most recent first. Lighter payload ŌĆö no `steps` (avoids pulling every task's full trace just to render a list).

Response `200 OK`:

```json
[
  { "url": "http://localhost:8000/api/tasks/2/", "id": 2, "prompt": "weather in Toronto", "result": "Toronto: 18┬░C, Cloudy", "created_at": "2026-07-09T12:05:00Z" },
  { "url": "http://localhost:8000/api/tasks/1/", "id": 1, "prompt": "What is 15 + 27?", "result": "42", "created_at": "2026-07-09T12:00:00Z" }
]
```

## `GET /api/tasks/{id}/`

Fetch one task with its full execution trace ŌĆö used when a history row is clicked.

Response `200 OK`: same shape as the `POST` response above (includes `steps`).

Response `404 Not Found` ŌĆö standard DRF detail shape:

```json
{ "detail": "No Task matches the given query." }
```

`PUT`/`PATCH`/`DELETE` are not exposed on `/api/tasks/{id}/` ŌĆö the frontend never edits or deletes a task, so those verbs return `405`. There is no `/api/steps/` endpoint either; steps are only ever read as part of a task's `steps` array, never addressed on their own.

## Execution step shape

Matches this format:

```
Step 1: Received input "Convert to uppercase"
Step 2: Selected tool: TextProcessorTool
Step 3: Tool result: HELLO
Step 4: Returning result to user
```

`tool_name` is `null` on bookkeeping steps (received input / returning result) and set on tool-selection and tool-result steps. The frontend's `ExecutionTrace` component uses this to visually distinguish tool-execution steps from bookkeeping ones ŌĆö see `frontend/IMPLEMENTATION.md`.

Multi-step prompts (e.g. "What is 2 + 2 and weather in Toronto") produce one selected-tool/tool-result pair per matched sub-prompt, all under the same `Task`, still ordered by `step_number`.

## OpenAPI spec

Auto-generated by DRF's built-in `rest_framework.schemas.get_schema_view` (introspecting the router/viewsets/serializers above) rather than hand-written, so it can't drift out of sync with the code. Served as JSON at `/api/openapi.json` and rendered via Swagger UI on the homepage (`/`), below a header listing the available agent tools and an admin-login link. Requires the `inflection` and `uritemplate` packages (both in `requirements.txt`) ŌĆö DRF's schema generator hard-depends on them for operation IDs and path parameters.
